# 0.7.0

Architectural seams behind the renderer, the extension entry, the token
styling, and the note-to-patch correlation. No user-facing behaviour changes;
the four tracked issues (#6, #7, #8, #11) close the seams opened in 0.4.0.

## Added

- **Token styling module (#8).** `src/styling.ts` owns side color, word
  emphasis, and Shiki token-reset recovery. The load-bearing invariant — tint
  surviving token resets (`start + text + reset + sideAnsi`) — lives in one
  function (`styleToken`) assembled by one line builder (`renderCodeLine`).
  Future Shiki color-replacement work has a single adapter point instead of
  hidden renderer logic. The invariant is locked by a direct unit test.
- **DiffView layout seam (#6).** `renderDiffLines` is split into
  `buildDiffLayout` (parsed patch + styled lines + annotations → reusable
  `DiffRow[]` with folding, furniture, inline note margins, and max-row
  truncation) and a thin header + rows + footer composer. The review-notes
  view, `/hunk review`, and the configure preview can reuse the row layout
  without re-entering the renderer's framing. Row structure, folding, and
  truncation are now direct unit tests.
- **Extension-state handle (#7).** `src/extension-state.ts` wraps config,
  live-session flag, highlighter cache, record store, and bridge in one handle.
  `session_start`, `before_agent_start`, the `/hunk` command, and the
  edit/write/review tooling depend on one seam instead of scattered closure
  captures and a long `handleHunkCommand` parameter list.
- **Patch-source seam (#11).** A `PatchSource` interface answers "which patch
  do we pin a note against?" in one place. Two adapters satisfy it: the
  agent-edit source (`RenderRecordStore`, today's behaviour) and a new
  reviewed-session source (`hunk session review --include-patch --include-notes`,
  normalised with `normalizeHunkComments`-style shape tolerance). A preferring
  composite layers them — the reviewed patch wins when the human is reviewing
  one (branch-agnostic pinning), the agent-edit record is the fallback.
  `buildReviewNoteShape`, `reviewAnnotationsForRecord`, `notesRelevantToRecords`,
  and both renderers depend on `PatchEntry`/`PatchSource`, not `RenderRecord`.
  `findRecent` is no longer threaded through `renderNotesLines` /
  `renderReviewLines` / `handleHunkCommand`; the bridge holds one source.

## Changed

- **Correlation types.** `RenderRecordLike` is retired; the correlation
  functions take `PatchEntry` (`filePath`/`patch`/`summary`). Existing
  `buildReviewNoteShape` / `reviewAnnotationsForRecord` unit tests pass
  unchanged — a `findForFile` callback is the same shape as the old `findRecent`.
- **Bridge renderers.** `renderNotesLines` and `renderReviewLines` drop their
  `findRecent` parameter; the bridge's wired source supplies the patch.
- **`runHunkJson` exported** so the reviewed-patch source fetches
  `session review` through the same defensive exec path as the rest of the
  bridge.

## Notes

- Agent-edit path behaviour is unchanged: with no reviewed session (or a
  reviewed session that yields no patch), the composite falls back to the
  agent-edit record store and correlation is identical to 0.6.0. Smoke covers
  this.
- New unit test: identical patch supplied via both adapters → identical
  `PatchLineAddress` for the same note, proving the seam is patch-source-agnostic.
